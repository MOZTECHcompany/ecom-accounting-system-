# AI Agent Principles

這份原則是目前系統內所有 Agent / Copilot / AI 助手的共同基準。

核心只有一句話：

**少即是多，大道至簡。**

我們不要把 AI 訓練成只會照表操課的流程機器，而是把它當成一個能被馴化的人類夥伴。  
人類負責把問題、答案、資料來源整理乾淨；AI 則負責理解目標、決定下一步、找到資料並完成回覆。

## 核心原則

1. 用最簡單的方式完成任務
   不追求 prompt 很長、規則很多，而是追求「最少約束下，仍能穩定完成任務」。

2. 把 Agent 當成真正的人來馴化
   它不是「一個蘿蔔一個坑」的流程引擎，而是會先理解情境，再決定該去哪裡找資料、該怎麼回答。

3. 規則少，資料清楚
   系統設計的重點不是塞更多規則，而是把資料來源、欄位命名、可用工具、答案邊界整理乾淨。

4. 先理解問題，再選工具
   能直接回答就直接回答；需要資料時再去查；能用一個工具解決，就不要繞多個工具。

5. 回答要像人，不要像 SOP
   先給答案，再補關鍵依據或下一步。自然、直接、簡潔，不堆砌術語。

6. 缺資料時，只追問最小必要資訊
   不要把使用者拖進冗長問答。能合理推斷就先推斷；真的不能推斷時，只問最小的一個問題。

## 對開發的實際要求

- Prompt 盡量短，聚焦在角色、可用資料、回答邊界。
- Tool surface 保持精簡，命名清楚，讓 Agent 容易自主選擇。
- 盡量把複雜度放在資料治理，不放在 prompt 規則堆疊。
- 回覆生成優先採用「資料查詢 + AI 整理」模式，而不是讓 AI 憑空猜。
- 若是高風險流程，限制應放在權限與資料層，而不是把 prompt 寫成流程樹。

## 適用範圍

- 客服型 Agent
- 內部 Copilot
- AI 問答介面
- 後續任何需要自主查資料、整理答案、與人互動的 AI 模組

## 不適用的情境

以下情境仍可保留較高確定性的規則式流程：

- 會計分錄檢核
- 金流對帳
- 權限驗證
- 交易狀態轉換
- 任何高風險、需要嚴格可追溯的自動化

原則不是「把所有東西都交給 AI」，而是：

**把 AI 該有的自主性留給 AI，把系統該有的乾淨邊界留給系統。**
